## M1 — Contora v0.7 «Ноль рутины» (июль, ~12–15 dev-days)

Цель: убрать ручную рутину из ежедневного флоу; сделать Contora рабочим инструментом калибровочных интервью и созвонов команды IMS к августу.

### F1. Zoom multi-track import + watch folder (4–5 дней) — якорная фича

**UX.**
- Settings → Sources: список наблюдаемых папок (дефолт `%USERPROFILE%\Documents\Zoom`, выключено до явного включения), переключатель on/off per папка.
- При появлении завершённой записи — InfoBar/toast: «Обнаружена запись Zoom „Планёрка IMS" (42 мин, 4 участника). [Импортировать] [Игнорировать]». Никакого автоимпорта по умолчанию (настройка «импортировать автоматически» — опционально, off).
- Импортированная сессия: название из имени папки Zoom, спикеры с реальными именами, диаризация не выполнялась.

**Техдизайн.**
- Новая папка `src/AudioRecorder.Services/Sources/`:
  - `WatchFolderService` — `FileSystemWatcher` + периодический скан (recovery после сна/оффлайна). Готовность записи определять по стабильности размера файлов (Zoom дописывает файлы после конца встречи): файл «готов», если размер не менялся N секунд и нет lock.
  - `ZoomRecordingDetector` — распознаёт структуру `<root>\<yyyy-MM-dd hh.mm.ss название>\Audio Record\*.m4a` (имя участника в имени файла) и fallback-структуры (одиночный `audio_only.m4a` без multi-track → обычный импорт с диаризацией). Результат — `DetectedRecording { Title, StartedAt, Tracks[{ParticipantName, Path}] }`.
  - `MultiTrackImportService` — оркестрация: m4a → wav (ffmpeg уже есть в пайплайне), VAD/RMS-проверка каждого трека (пустые дорожки не гонять через GPU), транскрипция каждого трека существующим `ITranscriptionService` (whisper-net), присвоение `Speaker = ParticipantName`.
  - `MultiTrackTranscriptMerger` — слияние сегментов всех дорожек по таймкодам в единый транскрипт (сортировка по start; пересечения допустимы — это живая речь).
- Модель `Session`: новые поля `SourceType` (`recorded | imported | zoom-multitrack | imported-vtt`), `ParticipantsJson`. Миграция SQLite — `ALTER TABLE ADD COLUMN` по образцу существующих.
- Диаризация для multi-track пропускается целиком (branch в пайплайне до `CreateDiarizationService()`).

**Риски/краевые случаи:** запись ещё идёт (стабильность файлов); тёзки в имени файлов (суффиксовать); очень длинные встречи (по трекам транскрипция даже быстрее — меньше пауз после VAD-отсечки); у пользователя не включена опция Zoom (детектор видит только mixed-файл → обычный импорт + подсказка в UI «включи Record separate audio…» со ссылкой на инструкцию).

**Приёмка:** реальная запись планёрки IMS в Zoom импортируется одним кликом; в транскрипте реальные имена без единого ручного действия; сессия с одиночным audio-файлом импортируется по старому пути.

### F2. Meeting templates (2–3 дня)

**UX.** Settings → Templates: CRUD шаблонов. Поля: имя, маска названия (`Планёрка IMS — {date}`), проект (для vault/Outline-коллекции), ожидаемые спикеры, LLM-промпт (пресет или свой), язык, авто-транскрипция (off/ask/on), авто-экспорт. При старте записи и при импорте — компактный выбор шаблона (дефолт «без шаблона»); опциональная подсказка «сейчас время планёрки IMS?» по дню недели/времени (эвристика, без календарей).

**Техдизайн.** Модель `MeetingTemplate` в Core; таблица `meeting_templates` в `SqliteSessionStore` (или отдельный `TemplateStore`); поле `Session.TemplateId`. Применение шаблона: название, prefill `SpeakerNamesJson`, промпт в `SessionPipelineService` (сейчас промпт один — параметризовать `SessionPipelineOptions`).

**Приёмка:** запуск записи планёрки = один клик по шаблону; название, спикеры, промпт и маршрут экспорта подставлены.

### F3. Speaker profiles v1 (1–2 дня) — роадмап #6

**Техдизайн.** Таблица `speaker_profiles(id, canonical_name, created_at)`; при ручном переименовании спикера в сессии — чекбокс «запомнить». Применение: подсказки автодополнения при переименовании + prefill из шаблона (F2). UI: Settings → Speakers (список, merge, delete). Поле `speaker_profiles.voice_embedding BLOB NULL` заложить сразу (заполняется в M3, миграция не понадобится).

### F4. Очередь авто-транскрипции (1–2 дня)

**Техдизайн.** `TranscriptionQueueService`: FIFO, одна задача на GPU (правило v0.4.1 сохраняется), enqueue по событиям «запись остановлена» / «импорт завершён» согласно настройке (off/ask/on; ask = текущее поведение, остаётся дефолтом). UI: индикатор очереди в списке сессий (`Queued` как под-состояние `Recorded` — без изменения enum `SessionState`, флаг в отдельном поле/памяти сервиса).

**Приёмка:** остановил запись → ничего не нажимая, через N минут сессия `Transcribed` (при включённом «on»).

### F5. VTT import (2 дня)

**UX.** При импорте аудио автоматически подхватывать одноимённый/соседний `.vtt` (файлы облачных записей Zoom): «Найден транскрипт Zoom с именами спикеров. Использовать имена из него? [Да/Нет]». Отдельный пункт «Импортировать аудио + VTT…».

**Техдизайн.** `VttParser` (cue: start/end/`Имя: текст`); собственная Whisper-транскрипция как обычно; наложение имён — переиспользовать логику `TranscriptionDiarizationMerger` (majority-overlap по таймкодам): VTT-cues выступают в роли «диаризационных» сегментов с готовыми именами. При расхождении таймлайнов (>2 c дрейфа) — предупреждение, fallback на обычную диаризацию.

**Порядок в M1:** F3 → F2 → F4 → F1 → F5 (мелкое раньше — быстрые победы и данные для F1; F1 — самая большая).
**Релиз v0.7.0** + пост build in public («как мы убили ручную разметку спикеров»).

## M2 — Contora v0.8 «Context Vault + мост IMS» (к 01.08 частично, полностью к 01.09; ~10–12 dev-days)

Цель: транскрипт перестаёт быть файлом «где-то в Outline» и становится структурированной памятью — локально (self-hosted трек, 01.08) и на сервере агента (бот v1.0, 01.09).

### F6. Согласование vault-схемы (не код; блокер, дедлайн ~15.07)

Совместно с Султаном зафиксировать **общий формат файловой памяти** (Hermes-стек, паттерн LLM Wiki):
- структура папок: `vault/<project>/meetings/<YYYY-MM-DD>-<slug>.md`; `index.md` per проект;
- frontmatter v1: `title, date, project, participants[], tags[], source (contora|dictator|manual), duration, summary, action_items[], decisions[]`;
- правила слаггирования, кодировка, максимальные размеры, конвенция для «сырого транскрипта vs выжимки» (предложение: выжимка + фронтматтер в основном файле, полный транскрипт — рядом `*.transcript.md`, чтобы не раздувать контекст агента);
- версионирование схемы (`schema: 1` в фронтматтере).
Результат — контракт в `shared/contracts/context-vault-contract.md` (по образцу transcription-server-contract).

### F7. VaultExportService (3 дня)

**Техдизайн.** `src/AudioRecorder.Services/Integrations/VaultExportService.cs`: запись md по контракту F6, регенерация `index.md`, настройка корня vault (дефолт `%USERPROFILE%\Documents\Contora\vault`). Автоэкспорт после LLM-пайплайна согласно шаблону (F2). Поле `Session.VaultPath`.

**Приёмка:** после транскрипции планёрки файл появляется в vault; Claude Code, открытый в папке vault, отвечает на вопросы по последней встрече без каких-либо ручных перекладываний. (Это же — демо для поста к self-hosted агенту 01.08.)

### F8. Экспорт-адаптеры (рефакторинг, 2 дня)

`IExportAdapter { Name, ExportAsync(Session, TranscriptDocument) }`; реализации: `VaultExportAdapter` (F7), `OutlineExportAdapter` (обёртка над существующим `OutlineService`), `ClipboardExportAdapter`. Маршрутизация по шаблону встречи. Outline становится опцией, а не обязательным шагом — текущее поведение сохраняется дефолтом для существующих настроек.

### F9. Sense Engine ingest (2–3 дня, после решения по транспорту)

**Вариант A (папка/синк):** vault лежит в git/синкается на сервер агента; Contora ничего не знает про сервер. Плюс: ноль кода. Минус: ручная настройка.
**Вариант B (HTTP):** endpoint на сервере агента (`POST /ingest/document`, Bearer token — аналогично Outline API); `SenseEngineExportAdapter` в Contora (config: URL + token + mapping проект→клиент для агентского трека, по образцу существующего Outline mapping; без хардкода URL в репозитории — правило из v0.4.2 сохраняется).
Рекомендация: начать с A для self-hosted (01.08), добавить B к боту v1.0 (01.09). Решение за Султаном+Никитой.

### F10. Раздельные mic/system дорожки (3 дня)

**Техдизайн.** `WasapiAudioCaptureService`: писать loopback и mic в **два** wav (сейчас микшируются). Пайплайн: mic-дорожка транскрибируется без диаризации (`Speaker` = имя владельца из настроек), system — с диаризацией; merge через `MultiTrackTranscriptMerger` (F1). После транскрипции — конвертация в один mp3 для хранения (текущее поведение хранения не меняется). Фича-флаг `SeparateTracks` (дефолт on после обкатки, до этого off).
**Выигрыш:** «я» всегда назван; диаризация решает задачу на N−1 спикеров; заодно исчезает класс багов с балансом громкости mic/system в миксе.

**Релиз v0.8.0** (можно дробно: v0.7.5 с F7 к 01.08).

## M3 — Contora v0.9 «Автономная разметка» + Dictator (сентябрь, ~8–10 dev-days)

### F11. Speaker ID v2 — голосовые эмбеддинги (3–4 дня, исследовательская)

`SherpaOnnxDiarizationService` уже вычисляет CAM++ эмбеддинги (3D-Speaker) для кластеризации — сейчас они выбрасываются. План:
- при ручном именовании спикера сохранять центроид кластера в `speaker_profiles.voice_embedding` (скользящее среднее по сессиям);
- на новых сессиях после диаризации матчить кластеры к профилям (cosine similarity); выше порога → автоимя с бейджем «авто: Иван, 87% [подтвердить/исправить]»; подтверждение уточняет эталон;
- порог подобрать на своём архиве записей (несколько десятков сессий с известной разметкой — готовый тестовый датасет); старт ~0.6, метрика: precision > 0.95 (ложное имя хуже, чем SPEAKER_00);
- для Sortformer-пути — прогнать sherpa-embedding extractor по сегментам кластеров отдельным шагом (модель уже в рантайме).
Ограничение: не работает для multi-track импорта (там и не нужно) и плохо — для спикеров < 10 сек речи (не матчить, оставлять SPEAKER_XX).

### F12. Zoom Cloud API коннектор (2–3 дня)

OAuth-app (user-managed), **polling** `GET /users/me/recordings` при старте приложения и по таймеру — webhook не нужен для single-user, не требует публичного endpoint. Новые записи → скачивание audio + VTT → пайплайн F5. Settings → Sources → Zoom Cloud (Client ID/Secret, статус подключения). Токены — в `settings.json` с DPAPI-шифрованием.

### F13. Dictator → Sense Engine (2 дня, в репозитории Dictator)

Действие «Отправить агенту» для расшифрованной диктовки: HTTP ingest (транспорт F9-B) с типом `note` — «надиктовал мысль → она в памяти агента» (сцена 1 сайта IMS, но локальным инструментом). Плюс: пишем в vault (`vault/<project>/notes/`), если настроен, — тогда работает и в self-hosted сценарии без сервера.

**Релиз v0.9.0** + пост («приложение узнаёт коллег по голосу»).

## Главные риски плана

1. **Ёмкость**: июль-август перегружены запуском IMS. Митигация: M1 разбит на независимые мелкие фичи, каждая шипится отдельно; ничего в M1 не блокирует IMS.
2. **Vault-схема согласована поздно** → F7 пишет во «временный» формат и потом мигрирует. Митигация: жёсткий дедлайн 15.07, схема с `schema: 1` и версионированием с первого дня.
3. **Zoom меняет структуру папок/имена файлов** локальных записей. Митигация: детектор терпимый к вариациям + graceful fallback на обычный импорт.
4. **Voice-ID окажется ненадёжным** на реальных данных (шумные loopback-записи). Митигация: F11 — исследовательская, precision-first, всегда с ручным подтверждением; при провале остаётся v1-профили + multi-track путь.
