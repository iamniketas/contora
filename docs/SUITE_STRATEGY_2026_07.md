## 1. Резюме (TL;DR)

1. **Центр экосистемы — Sense Engine, а не сьют.** После разворота IMS в продуктовую компанию (планёрка 01.07) главный продукт — копайлот личного бренда. Contora и Dictator — не параллельный бизнес, а **слой захвата контекста** этой экосистемы: они превращают устную реальность (встречи, консультации, надиктованные мысли) в структурированный текст, которым питается агент.
2. **Сьют — «федерация», не комбайн.** Рынок подтверждает: выигрывают продукты одной боли с отличным UX (Granola, Wispr Flow). Каждое приложение решает одну задачу, связывает их общий слой: runtime, модели, спикер-профили, формат данных (vault).
3. **Ключевое связующее звено — Context Vault**: локальная структурированная Markdown-база (паттерн файловой памяти а-ля LLM Wiki — тот же, что у Sense Engine). Vault — контракт между инструментами и агентом; его схему нужно согласовать с Султаном **до** реализации.
4. **Никакого параллельного продуктового запуска сьюта до 1 октября.** IMS ведёт never-ending launch (15.07 → 01.10), он потребляет всю публичную и значительную часть инженерной ёмкости Никиты. Роль сьюта в этот период: (а) внутренний инструмент команды и агентского трека, (б) поставщик контента для build in public, (в) техническая синергия с self-hosted агентом (01.08).
5. **Краткосрочный приоритет разработки — «ноль рутины» в Contora**: Zoom multi-track import (имена спикеров без диаризации), шаблоны встреч, спикер-профили, авто-очередь. Это 3–4 недели работы, немедленная ценность для ежедневного использования и для работы команды IMS с клиентами.

## 2. Текущий флоу и его боли (as-is)

Реальный сценарий использования сегодня:

1. Встреча в Zoom → запись в Contora (system loopback + mic, **сливаются в один файл**).
2. Либо: Zoom сам сохраняет запись → вручную найти файл на диске → вручную импортировать.
3. Вручную запустить транскрипцию.
4. Диаризация даёт SPEAKER_00/01/02 → **вручную переименовать спикеров** (каждый раз, даже на повторяющихся планёрках с тем же составом).
5. **Вручную ввести название встречи** («Планёрка IMS такая-то»).
6. Экспорт в Outline → **скачать MD из Outline** → работать с ним локально (Claude Code и т.д.).

Пункты 2, 4, 5, 6 — чистая рутина, автоматизируемая без «навязчивого» автодетекта встреч.

Архитектурная боль: **слияние mic + system в один файл до транскрипции уничтожает бесплатную информацию о спикерах.** Микрофонная дорожка — это всегда «я»; слив её с системным звуком, мы заставляем диаризацию заново угадывать известное.

Технический контекст (обновлено): с v0.6.0 транскрипция по умолчанию — in-process **Whisper.net** (CUDA, ~13–17× realtime на large-v3), диаризация — **Sortformer** (Python-сервер) с fallback **sherpa-onnx** (pyannote-seg-3.0 + CAM++ эмбеддинги, полностью in-process). Легаси faster-whisper-xxl сохранён. Важно: CAM++ эмбеддинги уже вычисляются — голосовая идентификация спикеров (§3.4) требует только их сохранения и матчинга.

## 3. Решения болей (конкретика, приоритет №1)

### 3.1 Zoom multi-track import — «имена спикеров бесплатно»

Ключевая находка исследования:

- **Локальная запись Zoom** с опцией *"Record a separate audio file for each participant"* создаёт в `Documents\Zoom\<meeting>\Audio Record\` **отдельный M4A на каждого участника с его именем в имени файла**. Работает даже на бесплатном аккаунте.
- **Облачная запись Zoom** (платный аккаунт): API отдаёт VTT-транскрипт **с именами спикеров** и mixed-audio. Транскрипт Zoom среднего качества, но имена и таймкоды точные.

Что делает Contora:

**A. Watch folder.** Слежение за папкой (дефолт `%USERPROFILE%\Documents\Zoom`). Новая запись → ненавязчивый toast «Обнаружена запись Zoom „Планёрка IMS" (42 мин, 4 участника). [Импортировать]». Никакого автодетекта идущих встреч — только реакция на сохранённые файлы. При импорте каждый M4A транскрибируется **отдельно** (диаризация не нужна), имя спикера — из имени файла, сегменты мержатся по таймкодам, название сессии — из имени папки Zoom.

**B. VTT import.** Пара «аудио + VTT»: качественная Whisper-расшифровка + имена/границы реплик из VTT (alignment по таймкодам). Позже — коннектор к Zoom API, записи подтягиваются сами.

**C. Обобщение — концепция Sources.** Zoom не хардкодится: `LiveCapture` (WASAPI), `FileImport`, `WatchFolder` (Zoom, OBS, Teams, любая папка), `CloudConnector` (Zoom API). Contora остаётся независимой от сервиса созвонов — Zoom просто первый и самый ценный адаптер.

### 3.2 Раздельные дорожки mic/system при собственной записи

Хранить два трека до транскрипции: mic = владелец (имя известно без диаризации), system = остальные (диаризация упрощается на одного спикера). Транскрибировать раздельно, мержить по таймкодам; после транскрипции можно схлопывать в один MP3 для хранения.

### 3.3 Шаблоны встреч (Meeting Templates)

Для повторяющихся встреч («Планёрка IMS», «Сессия с психотерапевтом»): шаблон = маска названия (`Планёрка IMS — {date}`), проект vault/коллекция Outline, ожидаемый состав спикеров, LLM-промпт, язык, флаг авто-транскрипции. Выбор одним кликом при старте/импорте; опциональная подсказка по дню недели/времени.

### 3.4 Глобальные профили спикеров + голосовая идентификация

- **v1:** таблица `speaker_profiles`; переименование «SPEAKER_00 → Иван» запоминается и предлагается в новых сессиях.
- **v2:** voice embeddings — сохранять CAM++ эталоны голоса именованных спикеров (уже вычисляются в sherpa-onnx пайплайне) и автоматически матчить кластеры новых сессий (cosine similarity + порог). После 2–3 встреч планёрка размечается именами полностью автоматически, независимо от сервиса созвона.

### 3.5 Очередь авто-транскрипции

После остановки записи / импорта сессия автоматически встаёт в очередь (одна за раз, GPU), настройка «выкл / спрашивать / авто».

### 3.6 Локальный Context Vault вместо круга через Outline

Сейчас: Contora → Outline → скачать MD → работать локально. Убрать круг:

- **Экспорт сразу в локальную структурированную папку**: `~/Documents/Contora/vault/<project>/<YYYY-MM-DD>-<title>.md` + `index.md`. Markdown с фронтматтером (дата, участники, теги, action items) — формат, с которым Claude Code и Sense Engine работают напрямую.
- Outline — **опциональный sync-адаптер**, не единственный выход.
- **Vault-схема = схема файловой памяти Sense Engine** (Hermes-стек, паттерн LLM Wiki — см. продуктовое описание IMS §6). Это осознанное решение: один формат для памяти агента и для выгрузок инструментов. Согласовать с Султаном до реализации.

## 4. Место сьюта в стратегии IMS

Синхронизация с «IMS: бизнес-стратегия (июль 2026)» и «IMS: продуктовое описание».

### 4.1 Каких кирпичиков не хватает (мэппинг на бизнес-стратегию)

Sense Engine хранит и развивает контекст пользователя, но сам контекст надо откуда-то брать. Продуктовое описание называет источники: голосовые в Telegram, ссылки, ответы на интервью. Сьют добавляет самый жирный и самый труднодоступный источник — **устную профессиональную реальность**:

| Кирпичик | Кто закрывает | Статус |
|---|---|---|
| Идеи/мысли голосом → текст (сцена 1 сайта: «надиктуй голосовое о неделе») | Dictator (локально) / Telegram-бот (в облаке) | Dictator работает; нет моста «надиктовал → ушло агенту» |
| Встречи, консультации, выступления → структурированный транскрипт | Contora | Работает, но с ручной рутиной (§2) |
| Транскрипт → база знаний агента (сервер, файловая вики) | **Мост Contora vault ↔ Sense Engine** | Отсутствует — главный недостающий кирпичик |
| Калибровочные интервью и созвоны агентского трека → материалы для онбординга клиента | Contora как инструмент команды IMS | Технически возможно уже сейчас, неудобно без §3 |
| Обогащение базы клиента командой (man-in-the-middle, продуктовое описание §10) | Contora + vault + доступ команды | Отсутствует |

Вывод: **сьют — это intake-слой Sense Engine.** Онбординг платного клиента агентства «частично выполняет команда, выгружая в агента готовые материалы» — эти материалы и есть выход Contora.

### 4.2 Мэппинг на три трека компании

- **Маркетинг:** релизы Contora/Dictator — готовый материал для founders' comms Никиты («о компании и продукте»). Каждая фича из §3 — пост в духе never-ending launch: конкретика есть, скриншоты есть, выйдет в ближайшие месяцы. Отдельный сюжет — анти-нейрослоп-совместимость: инструменты собирают **фактуру** (сырой контекст), а не генерируют финальный контент — это буквально позиция продукта («агент доводит до предпоследнего этапа»).
- **Продукт:** vault-мост (§4.1) + синергия с self-hosted агентом (01.08): self-hosted пользователь с Claude Code/OpenClaw + Contora vault получает **полностью локальный пайплайн** «встреча → контекст → агент» без облака вообще. Это уникальное предложение для power-юзеров self-hosted трека (которые и есть ЦА этого этапа) и сильный пост.
- **Компания:** для инвесторского трека сьют — доказательство платформенности (не «обёртка над ChatGPT», а экосистема со своим intake-слоем и локальным inference).

### 4.3 Синхронизация с таймлайном IMS

| Дата IMS | Событие IMS | Что даёт сьют |
|---|---|---|
| 15.07 | Анонс компании + сайт + опросник | Ничего не требуется. Не отвлекаемся |
| 01.08 | Self-hosted агент (репозиторий) + пост | **Vault-схема согласована и реализована** → пост может упоминать связку «Contora → локальная память агента» |
| ~авг | Профисследование, калибровочные интервью | Contora с «ноль рутины» (§3) — рабочий инструмент интервью: запись, имена, транскрипт, vault |
| 01.09 | Бот v1.0, waitlist-пользователи | Мост vault → сервер агента для агентского трека (команда обогащает базы клиентов) |
| 01.10 | Бот v2.0 + подписка | Решение о продуктизации сьюта наружу (см. §7) |

Принцип: **инженерные майлстоуны сьюта подчинены контент- и продуктовым майлстоунам IMS**, а не собственному релизному циклу.

### 4.4 Ограничение

До 01.10 не запускать сьют как отдельный публичный продукт с воронкой: два never-ending launch параллельно одна команда не вытянет (bus factor = 1 на инженерной части). Публичные релизы Contora на GitHub продолжаются в текущем режиме (build in public, без воронки и монетизации).

## 5. Рынок и конкуренты

Два пересекающихся рынка:

### 5.1 Meeting notes / AI notetakers

| Продукт | Модель | Обработка | Цена | Примечание |
|---|---|---|---|---|
| Granola | закрытый, cloud | облако (без бота) | ~$18/мес | лидер UX, «notepad, не бот» |
| Otter / Fireflies / Fathom / tl;dv | cloud, бот в звонке | облако | $10–20/мес | бот всех смущает — анти-фича подтверждена рынком |
| **Hyprnote** | open source, local-first | on-device | freemium | ближайший идеологический конкурент, растёт |
| BB Recorder, Mumble и др. | local-first | on-device | разово/подписка | ниша «privacy by architecture» оформилась |

Выводы: (1) ниша local-first реальна и растёт (юристы, врачи, терапия, консалтинг, NDA); (2) отличие Contora — **пайплайн до структурированного контекста** (vault, action items, агент), а не «заметки со встречи»; (3) диаризация с реальными именами — слабое место всех local-first игроков, §3.1/3.4 дают преимущество.

### 5.2 Dictation

| Продукт | Обработка | Цена | Модель |
|---|---|---|---|
| Wispr Flow | облако | $12–15/мес, free 2000 слов/нед | подписка, команды |
| superwhisper | локально + облако | $8.49/мес или $249 lifetime | freemium + lifetime |
| MacWhisper | локально | €59 lifetime | разовая покупка |
| Handy | локально | бесплатно, OSS | донаты |

Паттерн монетизации устоялся: **локальное = free/lifetime, облачное = подписка.** Dictator конкурирует с бесплатным Handy — сам по себе не денежник; его роль — ежедневная привычка и вход в экосистему.

### 5.3 «Комбайны»

Прецеденты (superwhisper добавляет meetings, Hyprnote тянется к вики, screenpipe «пишет всё») показывают: пользователи покупают решение одной боли, а не платформу. Платформа продаётся существующим пользователям как расширение. **Продавать по отдельности, связывать под капотом.** Важно: ни один из конкурентов не имеет того, что есть у IMS — агента-потребителя контекста. У них транскрипт — конечный продукт; у нас — сырьё.

## 6. Оценка идеи сьюта: сильные и слабые стороны

### Сильные стороны
1. **Ты — целевой пользователь**; скорость итераций максимальная.
2. **Shared runtime уже частично работает** (общие модели Contora+Dictator; GGML-модели v0.6.0 переиспользуют стор Dictator) — у конкурентов каждое приложение тянет свои гигабайты.
3. **Local-first + приватность** — растущий тренд и единственная защита от гигантов: Zoom/Teams/Notion добавят облачные заметки всем, локальную обработку — нет.
4. **Sense Engine как потребитель контекста** — уникальная связка: у конкурентов транскрипт — финал, у нас — вход в живую стратегию личного бренда.
5. **Канал дистрибуции через IMS**: white-glove setup клиентам агентства; воронка IMS — часть продукта.

### Слабые стороны и риски
1. **Bus factor = 1** на инженерной части сьюта — главный аргумент за «федерацию» с общими модулями и против расширения scope.
2. **Продукт заточен под флоу Никиты** — до продуктизации нужны внешние пользователи (Султан, команда, клиенты агентского трека — они появляются естественно из таймлайна IMS).
3. **Windows + WinUI 3 против Mac-центричного рынка** — минус (меньше ранних последователей) и плюс (на Windows local-first ниша почти пуста); macOS-ветки уже развиваются параллельно.
4. **Облачный inference = биллинг и поддержка** — второй бизнес; не начинать до валидации (и до продуктового ценообразования самой IMS — открытый вопрос №3 их стратегии).
5. **Конфликт ёмкости с таймлайном IMS** (15.07–01.10) — снят решением §4.4: сьют работает на IMS, а не параллельно ей.

### Вердикт
Идея жизнеспособна при последовательности: **снять рутину (§3) → мост в Sense Engine (§4.1) → инструмент команды и self-hosted трека → решение о внешней продуктизации после 01.10.** Пропуск ступеней (сразу биллинг и сьют-бренд) — главный риск.

## 7. Архитектура экосистемы (to-be)

```
┌───────────────┐  ┌───────────────┐  ┌──────────────────────────┐
│    Contora    │  │   Dictator    │  │ Sense Engine (сервер IMS)│
│ (встречи →    │  │ (диктовка →   │  │ Telegram-бот + дашборд   │
│  контекст)    │  │  промпты/идеи)│  │ файловая вики-память     │
└──────┬────────┘  └──────┬────────┘  └──────────┬───────────────┘
       │   локальный runtime-слой               │ ingest
┌──────┴──────────────────┴────────┐            │
│  Shared Local Runtime            │            │
│  • модели (GGML/ONNX, общий стор)│            │
│  • транскрипция (Whisper.net)    │            │
│  • диаризация + CAM++ эмбеддинги │            │
│  • speaker profiles, словарь     │            │
│  • LLM post-processing (Ollama)  │            │
└──────┬───────────────────────────┘            │
       │                                        │
┌──────┴────────────────────────────────────────┴──────────────┐
│              Context Vault (Markdown + frontmatter)           │
│  ~/Documents/Contora/vault/<project>/…                        │
│  • схема = файловая память Sense Engine (LLM Wiki pattern)    │
│  • читается Claude Code / self-hosted агентом напрямую        │
│  • sync-адаптеры: Sense Engine ingest, Outline, Telegram      │
└───────────────────────────────────────────────────────────────┘
```

Принципы:
- **Одна задача — одно приложение.** Contora не диктовщик, Dictator не пишет встречи, Sense Engine не транскрибирует часовые записи (дорого в облаке — локальный intake выгоднее и приватнее).
- **Общее — вниз, в runtime.** Словарь из Dictator улучшает расшифровки Contora; спикер-профили общие.
- **Vault — контракт.** Sense Engine не интегрируется с приложениями напрямую — он читает vault (локально у self-hosted, через ingest на сервере у бота). Это развязывает разработку команд.
- Кросс-платформа через контракты (HTTP-сервер, форматы файлов) — платформо-независимы; macOS-ветки общаются с ядром по тем же контрактам.

## 8. Бизнес-модель (эскиз, горизонт после 01.10)

До 01.10 сьют не монетизируется (§4.4). Дальше — три слоя, согласованных с монетизацией IMS («платим за людей» + «платный бот»):

| Уровень | Что входит | Ориентир |
|---|---|---|
| **Free / OSS-friendly** | локальные модели, запись, транскрипция, vault; связка с self-hosted агентом | 0 |
| **Suite Pro** | облачный inference (скорость), sync словаря/профилей, приоритетные фичи | $8–12/мес; кандидат на **бандл с подпиской Sense Engine** («подписка IMS включает инструменты захвата контекста») |
| **Lifetime local-only** | все локальные фичи навсегда | $99–149 (паттерн superwhisper/MacWhisper) |
| **IMS White-Glove** | настройка системы сбора контекста клиенту агентства (Contora + шаблоны + vault + агент) | внутри агентского трека «платим за людей» |

Логика: подписка оправдана только реальными облачными издержками; локальное не требует подписки — честно, совпадает с рынком, отличает от Wispr Flow и усиливает анти-нейрослоп/privacy-позиционирование IMS.

### Чего не делать
- Не строить биллинг/аккаунты сьюта до 01.10 и до прояснения ценообразования бота IMS.
- Не делать автодетект идущих встреч с предложением записи (осознанная анти-фича).
- Не превращать Contora в заметочник/вики — структурирование это территория Sense Engine.
- Не запускать публичную воронку сьюта параллельно воронке IMS.
- Не начинать Linux до валидации Windows+macOS.
